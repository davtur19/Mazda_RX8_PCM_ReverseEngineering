1.2  RX8_93c56_ECU_IC420_Read.bin
     ─────────────────────────────────────────────────────────────────
     Size:      256 bytes = full ABLIC S-93C56C EEPROM contents
     Source:    Physical EEPROM read from an EU RX-8 ECU (IC420 = EEPROM IC)
     VIN:       JM1FE17394010254 (at offset 0x80, 17 chars)
                JM1 = Mazda Japan, FE = RX-8, 173 = EU spec, 94 = 2004 MY
     Model ID:  "13H.6.020" (at offset 0x62)

     Magic byte: 0x5A at offset 0x00 (NOT 0x55 as previously documented)
     Second byte: 0xFF at offset 0x01

     Adaptive learning region (0x20-0x61):
       Contains REAL learned data — values like 0x81/0x01, 0x80/0x01, 0x7E/0x01
       organized as word pairs. This ECU has been driven and adapted.
       Pattern: decreasing first byte (0x81→0x59) with constant second byte (0x01)
       suggests fuel trim corrections indexed by load/RPM breakpoints.

     Immobilizer region (0x02-0x09):
       Unique per-vehicle: D3 D7 1A EF 41 3D 15 00
       These are encrypted key codes for the transponder immobilizer system.

     Post-VIN data (0x91+):
       0x95: 0x2A (configuration byte?)
       0x9A: 0x03, 0x9B: 0x0D (counters?)
       0xFD: 0x0C, 0xFF: 0x30 (checksum/status bytes)

2.1  CODE SIMILARITY ANALYSIS (vs our target 60E1D400)
     ─────────────────────────────────────────────────────────────────
     The 60E0xxxx series and 60E1xxxx series represent different firmware
     generations with fundamentally different code layouts:

       60E0E500 vs 60E1D400:  Code  7.4% similar,  Cal 15.1% similar
       60E0E700 vs 60E1D400:  Code  7.5% similar,  Cal 15.2% similar
       60E0FB00 vs 60E1D400:  Code  7.4% similar,  Cal 15.2% similar
       60E0FC00 vs 60E1D400:  Code  7.4% similar,  Cal 15.2% similar
       60E1B900 vs 60E1D400:  Code 16.8% similar,  Cal 15.4% similar

     Key observations:
       • 60E1B900 (N3ZDEB) is the closest to our target (same 60E1 platform)
         with 16.8% code similarity — enough to share core architecture
         but different enough that function addresses will differ
       • All 60E0xxxx ROMs are ~7% code similar to 60E1D400 — effectively
         a completely different codebase, only sharing SH7055 boilerplate
       • Calibration regions are ~15% similar across all variants, consistent
         with shared table structures but different values


2.2  60E0FB00 vs 60E0FC00 (NEAR-IDENTICAL PAIR)
     ─────────────────────────────────────────────────────────────────
     These two ROMs share the same CalID (N3Z2EBWW) and differ by only
     43 bytes (0.008% of total). Differences are all in calibration data:

       0x00FFC-0x00FFE:  2 bytes  (code area boundary)
       0x02005:          1 byte   (Denso PN: "B" vs "C")
       0x6D316-0x6D35E:  3 bytes  (CalID metadata)
       0x728D5-0x72901: 18 bytes  (calibration table values)
       0x77B47-0x77CC6: 12 bytes  (calibration table values)

     These are almost certainly sequential calibration revisions —
     FB00 and FC00 share identical firmware code with minor tuning tweaks.

6. EEPROM STRUCTURE CORRECTION
================================================================================

CLARIFICATION: Two separate validation mechanisms exist:

     A) PHYSICAL EEPROM CONTENT:
        Byte 0x00: 0x5A  (both dumps) — EEPROM-level validity marker
        Byte 0x01: 0xFF  (both dumps)
        This is the raw data stored in the 93C56C chip.

     B) RAM MIRROR HANDSHAKE (firmware at ROM 0x450):
        The eeprom_read_validate function checks byte +8 of the RAM mirror
        (0xFFFFDFE4 + 8 = 0xFFFFDFEC) for 0x55, then writes 0xAA after
        consuming the data. This is a producer-consumer protocol:
          SPI read ISR → writes EEPROM data to mirror, sets flag = 0x55
          eeprom_read_validate → reads data, sets flag = 0xAA ("consumed")
        Confirmed by disassembly: cmp/eq #0x55 at 0x456, mov #0xAA at 0x472.

     These are DIFFERENT validation layers. The 0x55/0xAA in our pseudocode
     documentation correctly describes the RAM handshake. The 0x5A at byte 0
     of the physical EEPROM is a separate EEPROM-level integrity marker.

EEPROM LAYOUT (confirmed from 2 physical dumps):
     0x00-0x01:  Magic/validity (0x5A, 0xFF)
     0x02-0x09:  Immobilizer key data (unique per vehicle, encrypted)
     0x0A-0x1F:  Configuration flags and counters
     0x20-0x61:  Adaptive learning data (fuel trim corrections, word pairs)
                 Factory default = 0xFF/0x00 pairs
                 Learned = decreasing values with 0x01 second byte
     0x62-0x6B:  Model identifier string ("13H.6.0xx")
     0x6C-0x7F:  Reserved (0xFF fill)
     0x80-0x90:  Vehicle ID (VIN for USDM/EU, chassis# for JDM)
     0x91-0x9F:  Configuration and counters
     0xA0-0xEF:  Reserved (0xFF fill)
     0xF0-0xFF:  Status/checksum bytes

7. CROSS-VALIDATION RESULTS
================================================================================

The RAM capture provides LIMITED validation for our N3J1EL analysis because
it comes from a different firmware family (N3Z2EB). However, some structural
observations transfer:

  CONFIRMED:
    ✓ Warm-reset magic 0x5AA5A55A at 0xFFFFDFFC — same across all firmwares
    ✓ CalID string stored in RAM near 0xFFFFDF70
    ✓ General RAM layout: sensors around 0xFFFFA3xx, state around 0xFFFFA4xx-A9xx
    ✓ Float32 representation for engine parameters (RPM, AFR, temp, load)
    ✓ Engine-off state zeroes most live variables (expected behavior)
    ✓ Residual values persist after shutdown (810 RPM, 29°C, 13.4 AFR)

  NOT CONFIRMABLE:
    ✗ Specific addresses (0xFFFFA424=RPM etc.) — may differ between firmwares
    ✗ Table descriptor locations — calibration regions differ
    ✗ Function addresses — code differs ~93% between 60E0 and 60E1 families

  RECOMMENDED NEXT STEPS:
    1. Obtain a RAM capture from the actual N3J1EL firmware (60E1D400)
       via OBD-II SID 0x23 ReadMemoryByAddress
    2. Compare 60E1B900 (N3ZDEB, same 60E1 platform) in IDA to validate
       shared function addresses and RAM layout
    3. Use the EEPROM dumps to verify the eeprom_read_validate function's
       actual magic byte check (0x5A vs 0x55)
