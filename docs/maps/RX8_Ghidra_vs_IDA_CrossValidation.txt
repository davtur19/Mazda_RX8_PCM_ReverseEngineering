3. ★ CORRECTIONS NEEDED IN OUR ANALYSIS
─────────────────────────────────────────────────────────────────────────────

  ★★★ CAN MESSAGE IDs — WRONG IN OUR DOCUMENTATION ★★★
  ─────────────────────────────────────────────────────
  WE CLAIMED:  0xCA (periodic 300ms, engine status) and 0xDC (event-driven, faults)
  GHIDRA SHOWS: Standard Mazda CAN IDs (decimal-coded):
    TX from ECU:
      0x201 — Engine RPM (bytes 0-1) + vehicle speed (4-5) + throttle (6) + status (7)
      0x203 — Engine status pack
      0x215 — TX pack
      0x231 — TX setup with getAndPack
      0x240 — TX getAndPack
      0x250 — TX getAndPack
      0x251 — TX getAndPack
      0x41  — TX pack
      0x420 — TX getAndPack (MIL status for CAN confirmed)
      0x620 — TX pack
      0x630 — TX getAndPack
      0x650 — TX getAndPack

    RX to ECU:
      0x212 — RX main handler
      0x216 — RX with plausibility check, fault eval, timer
      0x430 — RX unpack
      0x47  — RX unpack
      0x4B0 — RX main
      0x4B1 — RX unpack
      0x4C0 — RX main

  VERDICT: Our 0xCA/0xDC claims were FABRICATED — no evidence in either
  firmware. These IDs don't appear in the Ghidra analysis at all.
  The actual Mazda CAN bus uses standard 11-bit IDs in the 0x200-0x650 range.
  → MUST CORRECT in all documentation files.


  ★★ DIAGNOSTIC SESSION CODE 0x85 — PARTIALLY CONFIRMED ★★
  ───────────────────────────────────────────────────────────
  Ghidra: setDiagMode85Bool, setDiagMode85Bool2, calculateDiagSessionConditional,
          getDiagSessionSubFunction, securityNotUnlocked
  → Session code 0x85 appears CONFIRMED by Ghidra function names.
  → Sessions 0x81 and 0x87 still unverifiable.


  ★ O2 SENSOR TERMINOLOGY — CLARIFICATION NEEDED ★
  ─────────────────────────────────────────────────
  Ghidra uses "Front O2" and "Rear O2" (pre-cat and post-cat sensors):
    getFrontO2Voltage_fp, getRearO2FilteredValue, calcRearO2FuelTrim,
    getRearO2FuelTrimMode22, convertRearO2VoltageForMode22
  Our analysis uses generic "O2 feedback" and "lambda" terminology.
  → Ghidra's naming is more precise: Rear O2 is the post-cat sensor
    used for catalyst efficiency monitoring and LTFT feedback.


  ★ OMP MAX VALUE = 60 — NEW DETAIL ★
  ────────────────────────────────────
  Ghidra decompilation shows: "if (60 < (uVar7 & 0xff)) { uVar7 = 60; }"
  This means OMP duty cycle is capped at 60/255 = ~23.5% maximum.
  Our OMP analysis didn't identify this hard cap.
  → UPDATE OMP documentation with max=60 cap.


  ★ "VVT" NAMING — POSSIBLY INCORRECT ★
  ──────────────────────────────────────
  Ghidra has NO functions named "VVT" (Variable Valve Timing).
  The RX-8 Wankel has NO valves — there's nothing to vary.
  What we called "VVT" might actually be:
    - Intake runner length control (variable intake geometry)
    - SSV (Secondary Shutter Valve) auxiliary control
    - EGR (Exhaust Gas Recirculation) timing
    - Unknown ECU output control
  Ghidra's vvt_control_calc at 0x1C0FA (in Emissions section) suggests
  it might be an emission-control related actuator, not actual VVT.
  → MARK as ⚠ UNCERTAIN in our documentation. The RX-8 does NOT have VVT.


4. INSIGHTS FROM GHIDRA NOT IN OUR ANALYSIS

  4.1) FUEL CUT ARCHITECTURE — MUCH MORE COMPLEX THAN DOCUMENTED
  ─────────────────────────────────────────────────────────────────
  Ghidra reveals 30+ fuel cut functions with a CENTRAL ARBITRATOR:

  arbitrateFuelCut (0xE2D8):
    - OR's together 13+ independent fuel cut request flags
    - Output is 2-bit per-rotor: bit 0 = rotor A cut, bit 1 = rotor B cut
    - This means fuel cut can be applied to INDIVIDUAL ROTORS independently

  Fuel cut sources identified:
    1. Rev limiter: calculateRevLimiterFuelCut, revLimitFuelCutInit
    2. Throttle lift-off: throttleLiftFuelCutSomething, throttleLiftCountersandConditions,
       calculateOffThrottleORFuelCutTimer, driverOffThrottleSetPrevLoop,
       throttleLiftInitStuff, throttleLiftoffRelated
    3. DSC (traction control): arbitrateDSCFuelCut, dscPerRotorCut,
       DSC_checkIfMode_x10/x20/x40/x80
    4. Load-based: calculateFuelCutLoadBased
    5. Vehicle speed + braking: vehicleSpeedAndBrakingFuelCut
    6. Speed-based: speedBasedFuelCutSomething
    7. Throttle plate: throttlePlateSomethingFuelCut
    8. Per-rotor level 1: setPerRotorLevel1FuelCut, updateFuelCutPerRotorCalType
    9. Plausibility: getPlausibilityFuelCutFuelResumeEnrichment
    10. Condition-based: fuelCutCondtionRelated, evaluateFuelCutCondtionUnknown
    11. Driver request: setFuelCutDriver
    12. Timer after reset: fuelCutTimerAfterReset

  perRotorFuelCut (0x21762):
    - Wraps fuel cut in getSR/setSR (interrupt protection)
    - Handles structure values 0, 6, 0xC, 0x12 (rotor timing events)
    - Calls fuelCutArbitration, fuelCutSomethingSomething, dscPerRotorCut

  → Our documentation only described rev limiter and decel fuel cut.
    The actual system is vastly more complex with per-rotor granularity.

  4.6) STAGED INJECTION — PRIMARY + SECONDARY INJECTORS
  ─────────────────────────────────────────────────────
  calcInjectorOnTime (0x313F0) decompilation reveals:
    - Primary and secondary injector split algorithm
    - Split based on fuel request vs. injector capacity limits
    - Three operating modes:
      a) Request < primary max: split proportionally
      b) Request > total capacity: use latency-compensated split
      c) Request > primary max but < total: secondary fills remainder
    - Each injector path independently calculates on-time from cc/rev
    - Divided by injector size calibration (flow rate) for time conversion
    - First-order filter on injection time with saturation
